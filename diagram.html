4.1 Fallback Process via Blockchain

1
TM-A detects local TEE unavailability
A health check or timeout indicates that TEE-A1 and TEE-A2 are unavailable or overloaded. TM-A cannot process the job locally and initiates the fallback.
2
TM-A calls delegateToAny() on the Smart Contract
TM-A sends a transaction to Ganache calling the smart contract. The job ID and file URL are registered on-chain.
smartContract.delegateToAny(jobId, fileUrl)
3
Smart Contract selects target manager & emits event
The contract scans registered authorized managers, picks TM-B, records the job, and emits a JobDelegated event — without exposing any internal IPs.
emit JobDelegated(jobId, targetManagerAddress, fileUrl)
4
TM-B receives the JobDelegated event
TM-B continuously listens for blockchain events filtered by its own address. Upon receiving the event it extracts the job ID and file URL from the on-chain log.
5
TM-B downloads the input file
Using the file URL from the event, TM-B fetches the input payload from the Azure Storage blob container.
6
TM-B dispatches the job to TEE-B1
The job runs inside the Intel SGX enclave — confidential and tamper-resistant.
7
Result recorded on-chain
TEE-B1 returns the result to TM-B, which calls submitResults(). Execution is stored immutably on the blockchain.
smartContract.submitResults(jobId, charCount, message)
